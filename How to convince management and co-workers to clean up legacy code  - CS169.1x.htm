How to convince management and co-workers to clean up legacy code?

0

I work for a company where a tool with legacy code is used by about 200 non software engineers everyday. The development team is very small and untrained. The tool breaks once in a while but the team works very hard to fix the issues. Although the tool works most of the time, its potential to evolve remains largely untapped. The development team is itself not very enthusiastic about cleaning up the code due to a limited budget. How do you explain the issue to management and other co workers?

You need to try to estimate:

  1. the resources (person hours) spent/wasted fixing problems with the legacy software
  2. the effort required to clean up/rewrite the legacy software

When you have such estimates, you can make a cost/benefit analysis of the effort to clean up/rewrite the legacy software and present this to management and fellow workers. In addition to person hours, the analysis can include potential benefits such as the improved ability to add new functionality to the software.

Another approach which can be very effective is to do a small demo (on your own time) of cleaning up a portion of the legacy code and present this demo to others. Working software, even with very limited functionality, often works better for communicating that a project is worth doing than any verbal or written argument. Such a demo project would also help estimate the effort required for the full clean-up project.

I am making the assumption that there is a legitimate case to be made for rewriting/cleaning up the legacy software. Otherwise, of course, it probably is not worth the trouble to do the cost/benefit analysis. I think in many cases the cost of a substantial overhaul of existing software can be recovered fairly quickly via improved productivity. But it is difficult to argue for dedicating resources to rewriting or cleaning up software when there are always many other competing priorities.

There is a concept called "technical debt" that will probably be taught later in this course. In short, the idea is that when a team writes weak code, they build up this "debt" which needs to be repaid somehow.

Like any debt, you can repay it all at once, or in installments. And, like any other debt, it is usually easier to repay in installments. So, the best practice is often to dedicate a few hours each week to refactoring - to safely cleaning up small areas of the code. If done right, this will slowly but surely improve the quality of the codebase, at relatively low cost.

One aspect you can look at is there is any potential liability. If there is a major snafu with the legacy code (e.g. It doesn't run on the latest hardware, major crash, ...) what will be the consequences? Legacy code is generally left alone because it just works, even if nobody knows anymore how it works. But if it doesn't really work...
